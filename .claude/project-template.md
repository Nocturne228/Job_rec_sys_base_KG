# 求职面试项目标准化模板

> 基于 JobRec_KG 项目整理过程总结。适用于算法工程师/推荐系统/数据挖掘岗位的求职作品集项目。
> 可直接作为 Agent prompt 或新项目的初始化指南使用。

---

## 一、项目组织规范

### 1.1 顶层目录结构

```
project/
├── README.md                    # 项目完整概述（Agent 入口，唯一主文档）
├── pyproject.toml               # Python 项目配置
├── main.py                      # 端到端演示入口
├── .gitignore                   # Python/IDE/OS/项目数据忽略规则
├── src/                         # 源代码
│   ├── __init__.py              # 包文档字符串
│   ├── config/                  # 配置层
│   ├── data/                    # 数据层
│   ├── [module_1]/              # 功能模块（按架构分层组织）
│   ├── [module_2]/
│   └── utils/                   # 工具层
├── data/        (README)        # 数据存储目录 — 含 README 描述期望格式
├── models/      (README)        # 模型权重目录 — 含 README 描述保存格式
├── results/     (README)        # 实验结果目录 — 含 README 描述输出结构
├── logs/        (README)        # 日志目录 — 含 README 描述日志类型
├── docs/                        # 设计文档（按编号排序）
│   ├── 01-[设计文档].md          # SE 全生命周期文档
│   ├── 02-[数据规格].md          # 数据需求规格
│   └── 03-[知识清单].md          # 面试知识速查
└── ref/                         # 参考文献（按编号排序）
    ├── 01-[参考文献总览].md       # 所有文献合并
    └── 02-[文献应用映射].md       # 模块→文献映射
```

### 1.2 目录文档化要求

每个数据目录（`data/`、`models/`、`results/`、`logs/`）必须有 README.md，说明：
- 该目录的用途
- 当前已有内容
- 期望的数据/文件格式
- Demo vs 生产环境的差异
- 数据来源和生成方式

### 1.3 文档命名规范

- 统一使用中文命名（面向中文面试场景）
- 编号前缀 `01-` `02-` `03-` 明确阅读顺序
- 禁止混用英文大写命名（如 `DATA_SPEC.md`）和中文命名
- README.md 作为项目唯一入口，不分散到多个概述文件

## 二、代码规范

### 2.1 Python 代码

- 模块/子包：全小写 + 下划线（`recall/`, `skill_coverage.py`）
- 类名：PascalCase（`LightGCN`, `SBERTRecall`）
- 函数/变量：snake_case（`evaluate_model()`, `user_embeddings`）
- 每个子包必须包含 `__init__.py`，含 `__all__` 导出列表和包文档字符串

### 2.2 代码审计清单

在项目完成或提交前，逐项检查：

**P0 — 阻断性 Bug**：
- [ ] 所有模块间的键/ID 是否一致（字符串大小写、命名约定）
- [ ] 关键计算逻辑是否正确（公式与标准定义一致）
- [ ] 条件分支是否覆盖了所有路径（空值、边界、异常）

**P1 — 文档与代码一致性**：
- [ ] 文档中描述的特征维度、参数值与代码实际值是否一致
- [ ] 文档中的公式与代码实现是否匹配
- [ ] 数据规模数字在多处是否统一

**P2 — 硬编码值与论证**：
- [ ] 每个关键参数是否有可解释的依据（实验/论文/业务先验）
- [ ] 是否存在捏造的演示数据（如 `hash() % 30` 生成假分数）
- [ ] 超参数默认值与实际调用值是否一致

**P3 — 死代码与质量**：
- [ ] 是否有定义了但从未调用的函数/方法
- [ ] 是否有重复实现（如两个文件中相同的辅助函数）
- [ ] 是否有未使用的导入
- [ ] 类型注解是否正确（特别是 Optional 值）

### 2.3 面试防追问检查

每个关键参数必须能口头解释依据。不可接受的理由：
- "我随便设的"
- "从别人代码里抄的"
- "试了几个值这个最好"（除非有 grid search 记录）

可接受的依据类型：
- 论文消融实验结论
- 网格搜索 + 验证集指标的量化记录
- 工业界共识值 + 场景适用性分析
- 业务先验 + 漏斗转化率推导

## 三、文档体系

### 3.1 四层文档结构

| 层级 | 文件 | 受众 | 内容 |
|------|------|------|------|
| L0 入口 | `README.md` | Agent/面试官 | 完整项目概述（架构/指标/决策/Q&A） |
| L1 设计 | `docs/01-软件工程设计文档.md` | 面试官/同行 | SE 全生命周期 + 参数依据 + 面试话术 |
| L2 数据 | `docs/02-数据需求规格.md` | 开发者 | 数据字段定义/获取方式/处理流程 |
| L3 复习 | `docs/03-面试知识清单.md` | 自己 | 面试前快速自检的知识点清单 |

### 3.2 README.md 标准格式

```markdown
# 项目名 — 一句话定位

> 核心技术栈 | 角色 | 时间 | 规模

## 1. 项目定位与架构
- 一句话说明输入输出
- 架构图（ASCII art）

## 2. 数据规模
- Demo 规模 vs 生产目标规模

## 3. 模块详解
- 每个模块：选型依据 + 核心公式 + 参数表（值+依据）

## 4. 指标体系
- 离线指标对比表
- 离线→线上映射
- 北极星指标选择理由

## 5. A/B Testing 设计
- 实验分组/持续时长/检验方法

## 6. 关键技术决策
- 决策对比表（选择 vs 弃用方案 + 原因）

## 7. 面试高频 Q&A
- 每个 Q 一句话核心应答

## 8. 性能目标/运行/局限/演进
- 延迟/吞吐量表
- 运行命令
- 局限与演进路线

## 9. 项目目录
- 完整文件树

## 参考文献
- 核心文献 + 完整文献索引路径
```

### 3.3 REFERENCE 文档（Agent 快速理解用）

每个项目的 `ref/` 目录应包含：
1. **参考文献总览**：所有文献按模块分类，标注核心/参考/前沿级别
2. **文献应用映射报告**：每个模块→具体应用了哪些文献、应用在哪行代码

## 四、Agent 协作配置

### 4.1 Memory 系统

```
.claude-ds/projects/<project>/memory/
├── MEMORY.md                   # 索引文件（自动加载）
├── project_overview.md         # 项目完整概览（Agent 快速理解）
└── feedback_*.md               # 用户反馈记录
```

Memory 更新规则：
- 代码或文档发生显著变更时同步更新 `project_overview.md`
- 用户纠正方法或确认做法时记录到 `feedback_*.md`
- MEMORY.md 每行 < 150 字符，只做索引

### 4.2 Hook 配置

在 `.claude/settings.local.json` 中配置 PostToolUse hook：
- matcher: `Write|Edit`
- 当变更涉及 `.py` `.md` `.tex` `.json` `.toml` `.yaml` 文件时
- 通过 `additionalContext` 提醒 Agent 检查是否需要更新 memory
- 过滤掉 `__pycache__`、`.git/`、`.pkl`、`.pt`、`.pdf` 等非源码文件

### 4.3 Skills 组织

```
.claude/skills/
├── [skill-1]/SKILL.md
├── [skill-2]/SKILL.md
└── ...
```

- 每个 skill 放在独立子目录
- `SKILL.md` 用标准 frontmatter 格式
- 删除顶层 `.claude/skill_*.md` 等旧版设计稿（保留标准 skills）

## 五、项目清理流程

对每个新项目或已有项目，按以下顺序执行：

### Round 1: 文件清理
1. 删除 `.DS_Store`、`__pycache__/`、空目录
2. 删除其他 agent 系统的配置文件（如 `.qwen/`、`.agents/`）
3. 删除旧版设计稿（已被标准 skills 或正式文档取代的）
4. 合并完全重复的文件（diff 验证无差异）

### Round 2: 文档整合
1. 识别文档冗余：相同主题的多个文件 → 合并
2. 识别文档错位：放在错误目录的文件 → 移动
3. 合并时保留最完整版本的独特内容
4. 统一命名规范，添加编号前缀

### Round 3: 代码审计
1. 执行 §2.2 的 P0-P3 检查清单
2. 修复所有阻断性 Bug
3. 为所有关键参数补充论证依据
4. 删除死代码和重复代码

### Round 4: 目录文档化
1. 为 `data/` `models/` `results/` `logs/` 写入 README
2. 补齐缺失的 `__init__.py`
3. 更新项目 README.md 中的目录树
4. 同步更新 memory

## 六、面试准备自检清单

在面试前，确认以下问题都能流利回答：

### 技术深度
- [ ] 每个模块为什么选这个技术而不是替代方案？
- [ ] 每个关键参数的具体数值和选择依据是什么？
- [ ] 模型的 Loss 函数是怎么优化的？公式能写出来吗？
- [ ] 系统端到端延迟是多少？各层分别多少？

### 工程能力
- [ ] 数据从哪来？规模多大？Demo 和生产的区别？
- [ ] 有哪些已知局限？下一步怎么改进？
- [ ] A/B 实验怎么设计？检验方法是什么？
- [ ] 北极星指标是什么？为什么选它？

### 项目表述
- [ ] 30 秒电梯演讲：一句话说清输入/输出/架构
- [ ] 1 分钟架构拆解：各层职责和技术选型
- [ ] "你最满意的项目"：能说出一个系统性问题解决过程
- [ ] 能坦然说出项目的不足和改进方向

## 七、Agent Prompt 模板

将以下 prompt 用于新项目的初始化 Agent：

```
你要帮助我按照以下标准整理一个求职面试项目：

## 项目结构
- README.md 作为唯一项目入口，包含架构/指标/决策/Q&A/目录树
- docs/ 下按 01- 02- 03- 编号排序文档
- ref/ 下按 01- 02- 编号排序参考文献
- data/ models/ results/ logs/ 目录各含 README 说明期望格式
- src/ 下每个子包含 __init__.py 和 __all__

## 命名规范
- Python: snake_case 模块, PascalCase 类
- 文档: 中文 + 编号前缀, 禁止混用英文大写命名
- 禁止 README.md 之外的顶层概述文件

## 代码质量
- 消除所有模块间键/ID 不匹配
- 每个关键参数有论证依据
- 删除死代码、重复代码、未使用导入
- 公式与标准定义一致

## 文档合并原则
- 同主题多文件 → 合并保留最完整版本的独特内容
- 完全重复文件 → diff 确认后删除
- 旧版设计稿 → 被正式文档取代的即可删除
- 其他 agent 系统的配置文件 → 删除

## 文档化
- 数据目录即使为空也要写 README 说明期望格式
- 区分 Demo 当前状态 vs 生产目标状态

请先全面阅读项目，然后执行上述标准化操作。
```
