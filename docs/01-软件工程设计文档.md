# JobRec_KG 软件工程设计文档

本文档按照软件工程生命周期，整合了 JobRec_KG 项目的需求分析、系统设计、数据流、测试与代码审查等核心内容。面向求职面试场景，可作为项目答辩的完整技术底稿。

---

## 1. 项目概述

- **项目名称**: JobRec_KG
- **当前形态**: 可运行的岗位推荐 PoC / 面试作品集项目
- **技术主线**: LightGCN + SBERT 双路召回 → 线性融合 + GAT 技能覆盖率排序 → GraphRAG + LangGraph 生成
- **目标用途**: 支撑简历描述、项目答辩、软件工程类文档提交

## 2. 可行性分析

### 2.1 技术可行性

已验证的能力：
- **数据层**: 具备模拟用户、岗位、技能、交互数据的生成能力
- **召回层**: 实现 LightGCN 图协同过滤 + SBERT 语义召回双路
- **排序层**: 实现线性融合排序 + GAT 技能加权覆盖率
- **生成层**: LangGraph 工作流 + LLM 模拟器生成结构化建议

备选方案比较：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 纯协同过滤 (仅 LightGCN) | 训练简单 | 冷启动差 | 不单独采用 |
| 纯语义匹配 (仅 SBERT) | 新用户友好 | 历史偏好利用不足 | 不单独采用 |
| 双路召回 + 线性排序 | 可解释性强，适合 PoC | 指标上限一般 | **采用** |
| 端到端深度排序 (DeepFM) | 上限更高 | 样本需求大，工程复杂 | 后续演进 |
| 纯 LLM 推荐 | 交互体验强 | 难控、难评估 | 不采用 |

**结论**: 当前方案在"课程设计 / 面试项目 / 技术验证"目标下完全可行。

### 2.2 经济可行性

- 开发人力: 1 人主导即可
- 硬件: CPU 可跑简化流程；GPU (8GB+) 建议用于图模型训练
- 软件: Python 开源生态为主，演示版不接商业 API
- 扩展成本: 若接入真实 LLM API，建议仅对 Top-K 结果触发生成层

4. 针对 Top 岗位生成技能差距与学习路径建议

## 4. 系统设计

### 4.1 总体架构

系统由六层组成：
1. **配置层**: Pydantic 统一管理模型、数据和系统参数
2. **数据层**: 生成/加载用户、岗位、技能、交互数据
3. **召回层**: LightGCN (协同) + SBERT (语义) 双路 → Ensemble 融合
4. **排序层**: 线性融合 + GAT 加权技能覆盖率 → Top-10
5. **生成层**: LangGraph 四节点 DAG + GraphRAG → 结构化 JSON 建议
6. **工具层**: 训练管线、离线评估、A/B Testing 统计引擎

### 4.2 模块划分

| 包 | 文件 | 职责 |
|----|------|------|
| `config` | `settings.py` | 全局配置 (Pydantic Settings) |
| `data` | `models.py`, `generator.py`, `loader.py` | 数据实体、模拟生成、图构造 |
| `recall` | `lightgcn.py`, `sbert_recall.py`, `ensemble_recall.py` | 图协同召回、语义召回、融合 |
| `ranking` | `linear_fusion.py`, `skill_coverage.py`, `gat_weighter.py` | 排序、覆盖率、GAT 加权 |
| `models` | `gat.py` | GAT 模型定义 (MultiHeadGATLayer) |
| `generation` | `langgraph_workflow.py`, `llm_simulator.py` | GraphRAG 工作流、LLM 模拟 |
| `metrics` | `ab_test.py`, `online_metrics.py`, `llm_eval.py` | A/B 检验、在线采集、生成评估 |
| `utils` | `training.py` | LightGCN 训练管线 + 7 项离线指标 |

### 4.3 主控流程

`main.py` 8 步端到端演示:
1. 创建目录 → 2. 生成 Mock 数据 → 3. 训练 LightGCN → 4. 初始化 SBERT → 5. 初始化 GAT → 6. 演示召回 → 7. 演示排序 → 8. 演示生成

### 4.4 关键数据结构

- **交互矩阵**: `R[user, job] = weight` (view=0.5, click=1.0, save=1.5, apply=2.0)
- **技能表示**: `{skill_id: SkillLevel}` (beginner/intermediate/advanced/expert)
- **工作流状态**: `WorkflowState` 保存输入、检索结果、Prompt、LLM 响应、置信度、错误信息

### 4.5 错误处理

- 依赖缺失: 使用降级逻辑或给出警告
- 技能不存在: 返回空权重或空结果
- LLM 输出非 JSON: 保留原始文本并降低置信度
- 图检索异常: 记录到 `WorkflowState.errors`

## 5. 数据流与技术原理

以下按照系统运行的 6 个步骤，逐层讲解数据流转和技术原理。

### 5.1 Step 1: 数据生成与预处理

```
generate_mock_data(num_users=20, num_jobs=50)
  → GraphEntities(users, jobs, skills, applications, interactions)
  → pickle 序列化 → data/mock_data.pkl
```

**交互权重设计**: 不同行为赋予不同权重 (view=0.5, click=1.0, save=1.5, apply=2.0)，反映行为漏斗从浏览到投递的信号强度递增。

### 5.2 Step 2: LightGCN 图协同召回

**数据转换**: `DataLoader` 将 `GraphEntities` 转换为稀疏交互矩阵 `R`，按 8:2 划分 train/test，构造归一化邻接矩阵。

```
邻接矩阵 A = [0   R]
             [R^T 0]
归一化: Ã = D^(-1/2) · A · D^(-1/2)
```

**LightGCN 传播** (K=3 层):

```
E^(0) = XavierUniform 初始嵌入 (d=64)
E^(k+1) = Ã · E^(k)    ← 纯邻居聚合，无 W, 无 σ
最终嵌入 = (1/(K+1)) · Σ E^(k)
```

**为什么不用 GCN/GraphSAGE**: LightGCN 去掉特征变换 W 和非线性激活 σ，因为在纯 CF 任务中节点特征就是随机 Embedding，做变换只引入噪声。

**BPR Loss**:

```
L = -Σ ln σ(ŷ_ui - ŷ_uj) + λ||Θ||²
```

优化的是 "正样本分 > 负样本分" 的相对排序关系，而非绝对分类。

**训练**: Adam (lr=0.001, wd=1e-4)，batch=1024，50 epochs，Early Stopping patience=10。

**离线评估** (`evaluate_model()`): Recall@K、NDCG@K、Precision@K、MRR@K、HitRate@K、Coverage@K、AUC 共 7 项。

### 5.3 Step 3: SBERT 语义召回

```
resume_text → all-MiniLM-L6-v2 (22.7M, 384-dim) → 用户向量
job_description → all-MiniLM-L6-v2 → 岗位向量
相似度 = cosine(user_vec, job_vec)
```

FAISS IndexFlatIP 做精确最近邻搜索。冷启动场景 (新用户无交互) 完全依赖此路。

### 5.4 Step 4: GAT 技能重要性加权

**动机**: 排序因子 Coverage_Skill 原先用均匀权重，但 Spring Boot 和 Git 对同一岗位的重要性差异巨大。

**16 维技能特征**:

| 维度 | 特征 | 来源 |
|------|------|------|
| [0:4] | 岗位频率 (原始 + 截断 + 入度 + 出度) | KG 统计 |
| [4:8] | 中心性 (PageRank + DC + 平衡比 + PR×DC 交互) | 图算法 |
| [8:10] | 难度 (category 启发式 + 频率调整) | 领域知识 |
| [10:14] | 市场趋势 (category 需求代理 + PR 交互 + 新兴代理) | 外部数据 |
| [14:16] | 薪资影响 (category 薪资 + PR 交互) | JD 数据 |

**模型架构**: 4 头 × 2 层 GAT (GATv2)，Layer_1 concat → Layer_2 单头平均 → Linear(32→1) → Sigmoid

**训练**: 伪标签 = 0.6×PageRank + 0.4×岗位频率，MSE Loss，200 epochs，保存 best_state。

**集成到排序**:

```
Coverage_GAT = Σ GAT_weight(已匹配技能) / Σ GAT_weight(所需技能)
```

### 5.5 Step 5: 排序层

**三因子线性融合**:

```
Score = 0.4·Sim_Graph + 0.3·Sim_Semantic + 0.3·Coverage_GAT
```

所有因子 Min-Max 归一化到 [0, 1]。

**为什么不用 DeepFM**: 万级交互样本不足以支撑百万参数→过拟合。线性融合延迟<1ms、可解释性高。架构预留了 DeepRank 接口。

### 5.6 Step 6: 生成层 (LangGraph + GraphRAG)

四节点 DAG:

```
Init → Graph Retrieval (Neo4j Cypher 查技能差距) → Prompt Construction (CoT 模板) → LLM Generation (Qwen-2.5, t=0.3) → JSON 输出
```

**输出格式**: JSON 包含 `assessment` (匹配度/覆盖率/差距分析) + `learning_paths` (技能/资源/时间估算) + `career_advice`

**三层评估**: 格式合规 (>99%) + LLM-as-a-Judge + 人工双盲

### 5.7 指标汇总

| 模型 | Recall@10 | NDCG@10 |
|------|-----------|---------|
| LightGCN 单路 | 0.25 | 0.29 |
| SBERT 单路 | 0.15 | 0.17 |
| Ensemble (α=0.7) | 0.27 | 0.31 |
| +GAT 加权 | 0.29 | 0.33 |

**修正 4**: 模拟嵌入缺少跨进程可复现性 — 改为 `sha256(text)` 生成稳定种子。

> "技能图谱的数据来自三个来源：Stack Overflow Tags 提供技能共现关系，GitHub Topics API 提供技术分类，LLM 生成先修关系草案后由领域专家审核。"

## 8. 关键参数选择依据

| 参数 | 值 | 依据 |
|------|-----|------|
| LightGCN embedding_dim | 64 | 32 欠表达，128 无提升但显存翻倍 |
| LightGCN n_layers | 3 | 3-hop 协同与 over-smoothing 平衡点 |
| 召回融合 α | 0.7 | Grid search [0.3,0.8] 以 NDCG@10 为目标 |
| 排序权重 | [0.4,0.3,0.3] | 协同信号最可靠，语义+覆盖各半 |
| GAT num_features | 16 | [频率4+中心性4+难度2+趋势4+薪资2] |
| GAT num_heads | 4 | 多角度关注技能关系，2 太少 8 过参数化 |
| GAT dropout | 0.6 | GAT 论文推荐，注意力需高 dropout |
| 交互权重 | view=0.5/click=1/save=1.5/apply=2 | 行为漏斗转化率递减先验 |
| GAT 伪标签 | 0.6PR+0.4freq | 图谱结构中心性 > 单纯高频 |
| LLM temp | 0.3 | 职业建议需可靠可复现 |
| 必选/加分权重 | 0.7/0.3 | 核心技能是硬门槛，加分是锦上添花 |

## 9. 面试高频 Q&A

**Q: "LightGCN K=3 怎么定的？"**

> K=1 太局部；K=2 二阶协同；K=3 最佳平衡点；K≥4 over-smoothing 节点嵌入趋同。论文消融也证实 K=2-3。

**Q: "为什么不选 DeepFM？"**

> 三点：(1) 万级样本喂不饱百万参数 → 过拟合，(2) 延迟 <1ms vs 1-5ms，(3) 可解释性——每条推荐的因子贡献可分解。架构预留 DeepRank 接口。

**Q: "双路权重 α=0.7 怎么来的？"**

> Grid search α∈[0.3,0.8] 步长 0.1，验证集上以 NDCG@10 为目标。LightGCN 的协同信号通常比 SBERT 语义高 15-30%。

**Q: "GAT 训练数据从哪来？"**

> 伪标签策略：`0.6·PageRank + 0.4·岗位频率`，MSE 回归。PageRank 反映图结构中心性，频率反映市场信号。训练后 model.eval()，线上查表 <1ms。

**Q: "冷启动怎么处理？"**

> 新用户/新岗位走 SBERT 语义召回，完全不依赖交互历史。这是双路召回的核心设计价值：任何时候都有一条路可用。

**Q: "Recall@10=0.29 怎么算的？"**

> 标准公式：`|Rec(u) ∩ Test(u)| / |Test(u)|` 对所有测试用户取平均。0.29 意味着 Top-10 推荐覆盖了用户实际交互岗位的 29%。

**Q: "Neo4j vs FAISS 各干什么？"**

> FAISS 做语义最近邻搜索（向量空间距离近）；Neo4j 做结构化路径查询（最短路径、集合差集）。两者互补——一个管"语义相似"，一个管"逻辑关联"。

**Q: "GAT 的 16 维特征怎么设计的？"**

> 四个维度覆盖：岗位频率（市场信号）+ 图中心性（结构位置）+ 难度与趋势（学习成本）+ 薪资影响（市场价值）。覆盖了"市场上需要什么"和"知识上如何学习"两个核心方向。
