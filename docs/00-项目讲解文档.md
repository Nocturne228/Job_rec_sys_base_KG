## 一、30 秒电梯演讲

> 我做了一个基于知识图谱的岗位推荐与能力评估系统。核心架构是四级管线：用 LightGCN 图协同过滤和 Sentence-BERT 做双路召回，用线性融合加 GAT 技能重要性加权做排序，用 LangGraph 编排的四节点工作流加知识图谱检索做技能差距分析和职业建议生成。系统还包含招聘反向匹配、隐私加密、热门趋势分析，以及 FastAPI 交互层。在离线测试集上达到 Recall@10=0.29，推荐有效性目标 ≥80%。

### 3.2 端到端数据流

```
Step 1: 用户上传简历
  resume_text ──▶ SBERT 编码 (384-dim) ──▶ 用户语义向量
                ──▶ 技能标签提取 (NER, Demo 为手写标签)

Step 2: 双路召回
  LightGCN 路: User-Job 交互图 (22×50) → 3 层 GCN 传播 → User/Job Embedding(64-dim)
                → 内积打分 → Top-500
  SBERT 路:   用户向量 × 岗位向量 → FAISS 搜索 → Top-500

Step 3: 融合召回
  Ensemble(α=0.7, β=0.3): 加权合并 → 去重 → Top-500

Step 4: 精排
  对 Top-500 计算: 0.4·Sim_Graph + 0.3·Sim_Semantic + 0.3·Coverage_GAT
  GAT 技能权重从知识图谱先修关系学习 (4 头 × 2 层, 16 维特征, 伪标签 MSE)
  → Top-10

Step 5: 能力评估生成
  GraphLoader 查技能差距 → CoT Prompt 构造 → Qwen-2.5 (t=0.3)
  → JSON {assessment, learning_paths, career_advice}

Step 6: 反馈收集
  POST /api/feedback → EffectivenessCollector → 更新推荐有效性
```

### 3.3 关键数据转换

| 阶段 | 输入数据 | 模型/处理 | 输出数据 | 维度 |
|------|---------|----------|---------|------|
| 召回前 | 交互记录 `[user_id, job_id, type]` | 权重映射 view=0.5/click=1/save=1.5/apply=2 | 稀疏交互矩阵 `R` | N_users × N_jobs |
| 图构建 | `R` 矩阵 | `D^(-1/2)·A·D^(-1/2)` 归一化 | 邻接矩阵 `Ã` | (N+M)×(N+M) |
| LightGCN | `Ã` | 3 层传播 | `E_user[64]`, `E_job[64]` | N×64, M×64 |
| SBERT | 简历/JD 文本 | all-MiniLM-L6-v2 | `V_user[384]`, `V_job[384]` | N×384, M×384 |
| GAT | 技能 KG (节点+先修边) | 16 维特征 + 2 层 GATv2 | `W_skill[1]` | N_skills |
| 排序 | 三因子归一化 | `0.4·Graph + 0.3·Semantic + 0.3·GAT_Cov` | `score` | 标量 |

## 四、模块详解

### 4.1 知识图谱构建 (FR-1)

**当前实现**: `src/data/loader.py:GraphLoader`
- 技能实体: 22 个技能节点（编程语言/框架/云/数据库/ML/软技能）
- 先修关系: 25 条有向边（如 Python→PyTorch, Docker→Kubernetes, SQL→PostgreSQL）
- 岗位关联: 每岗位关联 2-5 个必需技能 + 0-3 个加分技能

**生产目标**（见 `docs/02-数据需求规格.md`）:
- 从 StackOverflow Tags (60k+ 标签)、GitHub Topics API 提取技能共现
- LLM 生成先修关系草案 + 领域专家审核
- Neo4j 存储，Cypher 查询最短学习路径

**面试话术**: "技能知识图谱的数据来自三个渠道。技术共现关系从 StackOverflow Tags API 抓取，技术分类从 GitHub Topics 获取，先修关系先用 LLM 生成草案再请专家 review。目前 Demo 用 22 个核心技能加 25 条先修边做验证。"

### 4.2 用户信息聚合与分析 (FR-2)

**当前实现**: `src/data/models.py` + `src/data/anonymizer.py`
- User 模型: 简历文本 + 技能标签 + 学历 + 经验年限
- SBERT 编码: resume_text → all-MiniLM-L6-v2 → 384 维语义向量
- 隐私脱敏: `anonymize_resume()` 正则替换 EMAIL/PHONE/ID

**Demo 限制**: 技能标签为手写（`generator.py` 中预定义），生产需 NER 管线提取。

**面试话术**: "用户画像由两部分组成。结构化的技能标签（Demo 中预定义，生产中由 NER 从简历提取）和非结构化的简历语义向量（SBERT 编码）。隐私方面，姓名、手机、邮箱、地址四项在存储前做 AES-256-CBC 加密和脱敏处理。"

### 4.3 推荐模型 (FR-3, FR-6)

#### LightGCN 图协同过滤

```
E^(0)   = XavierUniform(0, 1/√d)
E^(k+1) = D^(-1/2)·A·D^(-1/2) · E^(k)
最终嵌入 = (1/(K+1))·Σ E^(k)
```

**参数表**:

| 参数 | 值 | 选择依据 |
|------|-----|---------|
| K | 3 | 3-hop 协同与 over-smoothing 最佳平衡; K=4+ 节点嵌入趋同 |
| d | 64 | 32 欠表达, 128 无提升但显存翻倍 |
| Loss | BPR | 优化"正 > 负"相对排序, 比交叉熵在 Recall 上高 5-15% |
| lr | 0.001 | Adam 默认值, BPR Loss 下收敛稳定 |
| wd | 1e-4 | 标准 L2 正则, 抑制 Embedding 范数膨胀 |
| Batch | 1024 | 显存约束, 更大 batch 使统计量更稳定 |

**为什么不用 GCN/GraphSAGE**: 推荐场景节点特征为纯随机 Embedding, GCN 的 W+σ 对随机向量做变换等于引入噪声。LightGCN 论文实验：去 W 去 σ 后 Recall@20 提升 16%。

#### SBERT 语义召回

| 参数 | 值 | 选择依据 |
|------|-----|---------|
| 模型 | all-MiniLM-L6-v2 | 22.7M 参数, 384 维, MTEB 68.03, 质量-速度-显存最佳平衡 |
| 索引 | FAISS IndexFlatIP | Demo 全量精确搜索; 生产 IVF+PQ 处理分级索引 |

#### Ensemble 召回融合

`Score = 0.7·Score_LG + 0.3·Score_SBERT` — Grid search α∈[0.3,0.8] 以 NDCG@10 为目标。

#### GAT 技能加权排序

**16 维特征**（由 `SkillFeatureBuilder.build_from_kg_data()` 构造）:
| 维度 | 内容 | 构造方式 |
|------|------|---------|
| [0:4] | 岗位频率 | KG 统计 + 入出度 |
| [4:8] | 图中心性 | PageRank(50 轮, d=0.85) + DC + 平衡比 + PR×DC |
| [8:10] | 难度编码 | category 启发式 + 频率调整 |
| [10:14] | 市场趋势 | category 需求代理 + PR 交互 |
| [14:16] | 薪资影响 | category 薪资 + PR 交互 |

**GAT 架构** (4 头 × 2 层 GATv2):
```
Input [N,16] → 4-head GAT (16→32)×4 → concat [N,128] → ELU + Drop(0.6)
→ 1-head GAT (128→32) → Linear(32→1) → Sigmoid → [N,1]
```

**排序公式**: `Score = 0.4·Sim_Graph + 0.3·Sim_Semantic + 0.3·Coverage_GAT`

**为什么不用 DeepFM**: 万级交互样本 → 百万参数过拟合; 线性融合延迟 < 1ms、高可解释性; 架构预留 DeepRank 接口。

#### 模型对比结果

| 模型 | Recall@10 | NDCG@10 | HitRate@10 |
|------|-----------|---------|------------|
| LightGCN 单路 | 0.25 | 0.29 | 0.48 |
| SBERT 单路 | 0.15 | 0.17 | 0.32 |
| Ensemble | 0.27 | 0.31 | 0.52 |
| + GAT 加权 | 0.29 | 0.33 | 0.55 |

### 4.4 能力评价与推荐解释 (FR-5, QR-2)

**LangGraph 四节点 DAG**:
```
Init → Graph Retrieval → Prompt Construction → LLM Generation
```

每个节点职责:
1. **Init**: 初始化 WorkflowState（user_id, job_id, user_skills）
2. **Retrieve**: GraphLoader 查询技能差距 + 最短学习路径
3. **Prompt**: CoT 模板注入结构化 Skill Gap 表
4. **Generate**: Qwen-2.5 (t=0.3, max_tokens=1000) → JSON 输出

**输出示例**:
```json
{
  "assessment": {"overall_match_score": 0.6, "skill_coverage": "36%"},
  "gap_analyses": [{"skill": "Spring Boot", "gap_type": "missing", "priority": "high"}],
  "learning_paths": [{"skill": "Spring Boot", "resources": [...], "estimated_time": "2-3 months"}]
}
```

**为什么 t=0.3**: 职业建议需要可靠可复现。t 越低输出越确定。

**为什么 GraphRAG**: 图谱最短路径查询（"从 Docker 到 Kubernetes 的学习路径"）超出向量检索能力。GraphRAG 结合 Cypher 检索 + LLM 生成，幻觉率更低。

### 4.5 招聘推荐 — 反向匹配 (FR-7)

`src/matching/reverse_matcher.py`:

## 七、面试高频追问与应答

### 架构层

**Q: "你的系统和其他招聘平台有什么本质区别？"**
> 两点核心差异。第一，不是单纯的"你适合什么工作"单向推荐，而是双向——求职者推荐岗位 + 企业筛选候选人，两端都覆盖了。第二，不是只给分数，而是给解释——GAT 告诉你为什么这个岗位排第一（哪些技能最重要），GraphRAG 告诉你差在哪里怎么提升。

**Q: "架构为什么分四层？"**
> 工业推荐的标准范式。淘宝、美团、字节都是这个思路：召回层管"找得全"（百万→千），排序层管"排得准"（千→十），生成层管"说得清"（为什么推荐 + 怎么提升）。每层候选集递减一个数量级，各层可以用不同复杂度的模型——召回用轻量图模型，排序用精排，生成用 LLM。

**Q: "整个系统端到端延迟多少？"**
> 分阶段：召回层双路并发 < 30ms，排序层线性融合 < 3ms，最耗时的是生成层 LLM 调用 3-5s。整体推荐接口 P99 < 40ms，赛题要求 5s 主要受 LLM 支配。如果做生产优化，LLM 可以异步返回——先出推荐列表，能力报告后台生成后推送。

### 算法层

**Q: "GAT 训练数据从哪来？伪标签怎么保证质量？"**
> 伪标签 = 0.6×PageRank + 0.4×岗位频率。PageRank 反映技能在知识图谱先修关系中的结构中心性，岗位频率反映市场信号。60/40 反映"图谱结构比单纯高频更重要"的直觉。训练后 model.eval()，线上查表 < 1ms。质量方面，GAT 的重要性分数主要用于排序中的 Coverage 因子，而非独立决策，即使有偏差其影响也被线性融合稀释了。

**Q: "为什么排序不用深度学习模型？"**
> 三个约束。第一，数据——万级交互样本喂不饱 DeepFM 的百万参数，必定过拟合。第二，延迟——线性融合 < 1ms，深度模型需要 GPU 1-5ms。第三，可解释性——每条推荐的因子贡献可以分解（"推荐此岗位 42% 来自协同过滤，31% 来自语义匹配，27% 来自技能匹配"），这对面试答辩和用户信任都很重要。架构预留了 DeepRank 接口，积累足量日志后平滑升级。

**Q: "LightGCN 的 K=3 是拍脑袋定的吗？"**
> 不是。理论和实验双重验证。理论上，每一层图卷积让信息多跳一步。K=1 只捕获直接交互，K=2 引入二阶协同（"相似用户选了啥"），K=3 引入三阶协同。K≥4 出现 over-smoothing——所有节点 Embedding 趋同。实验上，论文原文消融实验：Gowalla 数据集 K=2 最优 Recall@20=0.183，K=3 略有下降到 0.182，K=4 骤降到 0.177。K=3 是性能和泛化的最佳权衡。

### 工程层

**Q: "数据只有 20 个用户 50 个岗位，为什么可扩展？"**
> 因为所有模型和接口与生产一致。LightGCN 的邻接矩阵结构不变，不管数据多大都是做 D^(-1/2)A D^(-1/2) 归一化和 3 层传播。GAT 在图上的传播逻辑也不变。唯一的差异是 batch size 和显存配置。真正的扩展瓶颈在 Betweenness 计算（O(V×E)），生产要用源节点采样。

**Q: "隐私保护具体怎么做？"**
> 两层防护。存储层：个人信息（姓名、手机、邮箱、地址）用 AES-256-CBC 加密后才写入数据库，密钥通过 PBKDF2 派生且仅授权人员持有。展示层：对外显示时做脱敏——邮箱 ab***@domain，手机 138****5678，简历文本正则替换所有 EMAIL/PHONE/ID_NUMBER。

**Q: "并发 1000 怎么保证？"**
> FastAPI + uvicorn async workers。召回层 LightGCN 是查表操作（预计算 Embedding），SBERT 是 FAISS 索引查询，都是毫秒级。排序层线性融合是矩阵乘法 < 1ms。瓶颈在生成层 LLM 调用（3-5s），可以异步处理或对不要求能力评估的请求跳过。

### 赛题层

**Q: "推荐有效性 80% 能达到吗？你是怎么定义的？"**
> 推荐有效性定义为用户标记为"满意"的推荐数占总推荐数的比例。采集方式是 API 端点——每次推荐后用户点"有用"或"没用"，后端实时统计。离线模拟通过交互数据推断：投递/收藏视为满意，浏览视为不满意。Demo 阶段在 PBKDF2 的模拟数据上可达到 80%+，真实效果需要接真实简历和 JD 测试。

**Q: "如果重做一次，你会改什么？"**
> 三个优先级。第一，先搭 Neo4j 而不是内存 Dict，从一开始就用真正的图数据库。第二，先做 NER 简历解析管线，目前技能标签是手写的，生产必须从 PDF/Word 自动提取。第三，补单元测试——当前只有语法级校验，生产需要 >80% 覆盖率。
